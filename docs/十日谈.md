# 十日谈 — "吉小图"开发历程

> DGX Spark Hackathon 参赛日记 · 吉利学院图书馆团队

---

## Day 0：方向选择 — "图书馆+AI"的天然契合

**7月12日 · 训练营启动日**

接到 Hackathon 题目时，团队第一反应是"做什么能既体现技术深度，又有真实落地价值？"

我们来自吉利学院图书馆，每天都在面对新生问路、找书、问规则。没有比这更真实的场景了——**图书馆本身就是一个天然的知识库+服务场景**。

方向一"图书馆新生AI导览助手"几乎没有争议就通过了。三个理由：
1. 我们懂这个场景（团队每天看着馆员重复回答同样的问题）
2. 我们有真实数据基础（馆藏数据库、规章制度、楼层平面图）
3. Demo极度直观（拍照→识别→导航，评委3秒看懂）

---

## Day 1-2：核心跑通 — 从零到第一个回答

**7月13-14日**

第一行代码是 `stepfun_client.py`——Step 3.7 Flash 是整个系统的发动机。花了整整一天调试 API 对接：OpenAI-compatible 协议不算难，但多模态输入格式踩了不少坑。图片 base64 编码、content 数组结构、temperature 参数选择……每一步都试错。

Day 2 晚上，第一个"你是谁"成功返回了正确回答。

> ⚠️ **关于数据**：正值暑期放假，图书馆处于闭馆状态，因此本项目数据目前为**测试数据集**，真实部署时可通过 MySQL 同步接口一键切换为生产数据。

---

## Day 3-4：Agent 架构 — 边界比能力更重要

**7月15-16日**

看了往届获奖作品（尤其是"Starfire-AgentTeam"）后，我们意识到评审想看的是**"组织化协作"**，不是"单兵作战"。

于是有了四个Agent的设计：
- Guide Agent → 只管对话和意图路由
- Search Agent → 只管RAG检索
- Planning Agent → 只做A*路径计算
- Verify Agent → 只做事实核查

每个Agent有明确的 `allowed_actions` 和 `forbidden_actions` 边界约束。

---

## Day 5：多模态 — 拍照秒识书架

**7月17日**

Step 3.7 Flash 的原生多模态能力远超预期。给出期刊阅览区照片，模型直接返回位置描述。不需要单独的 OCR、不需要 YOLO 检测——Flash 直接"看懂"了。

这是 Demo 的高光时刻：**拍照→识别→定位→回答**，一条链路打通。

---

## Day 6-8：本地大模型 — 把算力用起来

**7月18-20日**

DGX Spark GB10 拥有 128GB 统一内存，光用来调 API 太浪费了。

我们在云主机上编译了 llama.cpp（CUDA 13.0，sm_120 架构），从 ModelScope 国内源下载了 Qwen3.6-35B-A3B MoE 的 MXFP4 量化模型（~21GB）。模型加载 9 秒，推理速度约 43 tok/s。

最终采用混合路由架构：轻量任务（意图分类、校验）走 Stepfun API，重量任务（回复生成）走本地 GPU——充分利用平台算力，同时保证响应速度。

---

## Day 9-10：打磨 — 细节决定体验

**7月21-22日**

- 身份问题：本地 Qwen 模型默认回答"我是通义千问"，通过路由修复强制闲聊走 Stepfun，确保始终回应"我是吉小图"
- 拍照上传：相机权限问题改为文件上传，更稳定易用
- RAG 检索：嵌入模型因网络不可用，切换为 BM25 关键词匹配，每次从磁盘重载数据确保实时性
- IP 形象：设计了吉小图专属头像替换默认机器人图标
- 借阅规则：按学校实际政策更新（学生教师各20册，30天，7:30-22:00）

---

## 写在最后

十天时间，从一个想法到在 DGX Spark 上跑起来的完整多智能体系统。四个 Agent 各司其职，本地 GPU 和云端 API 协同工作，拍照识别和知识检索无缝衔接。

不管结果如何，这十天我们把"图书馆+AI"这件事，从想法变成了能跑、能看、能演示的作品。

**这就是 Hackathon 的意义。**
