> DGX Spark Hackathon 参赛日记 · 吉利学院图书馆团队

## Day 0：方向选择 — "图书馆+AI"的天然契合

**7月12日 · 训练营启动日**

接到 Hackathon 题目时，团队第一反应是"做什么能既体现技术深度，又有真实落地价值？"

我们来自吉利学院图书馆，每天都在面对新生问路、找书、问规则。没有比这更真实的场景了——**图书馆本身就是一个天然的知识库+服务场景**。

方向一"图书馆新生AI导览助手"几乎没有争议就通过了。三个理由：
1. 我们懂这个场景（团队每天看着馆员重复回答同样的问题）
2. 数据触手可及（馆藏MySQL数据库、规章制度、楼层平面图）
3. Demo极度直观（拍照→识别→导航，评委3秒看懂）

确定方向后，第一件事不是写代码，而是**激活DGX Spark云节点**——群里已经有人在排队了。还好我们手速快，当天就拿到了账号。

## Day 1-2：核心跑通 — 从一束光到一条线

**7月13-14日**

第一行代码不是Agent框架，而是 `stepfun_client.py`。

Step 3.7 Flash 是整个系统的发动机。如果发动机打不着火，造再漂亮的车也没用。我们花了整整一天调试API对接——OpenAI-compatible协议不算难，但多模态输入格式踩了不少坑。图片base64编码、content数组结构、temperature参数的选择……每一步都试错。

Day 2 晚上10点，第一个"你好，图书馆在哪"成功返回了正确回答。团队在群里发了三个🎉。

同一天，知识库数据同步启动。我们用 `pymysql` 连上了学校馆藏数据库（只读账号），500条真实图书记录被拉取到JSON文件中。**真实数据让这个项目的质感完全不同**——不再是一个"假设有书"的Demo，而是真正能帮学生找到"三体在2F文学区A-03书架"的系统。

## Day 3-4：Agent架构 — 边界比能力更重要

**7月15-16日**

这是整个项目最关键的阶段，也是我们最骄傲的设计决策。

最初的想法很简单：一个模型 + 一些 if-else = Agent。但看了往届获奖作品（尤其是"Starfire-AgentTeam"）后，我们意识到评审想看的是**"组织化协作"**，不是"单兵作战"。

于是有了四个Agent的设计：
- Guide Agent → 只管对话和意图路由
- Search Agent → 只管RAG检索
- Planning Agent → 只做A*路径计算
- Verify Agent → 只做事实核查

每个Agent有明确的 `allowed_actions` 和 `forbidden_actions`。Guide Agent **不能**直接查库，Search Agent **不能**对用户说话——这种严格的边界约束才是"多智能体系统"区别于"多模型堆叠"的本质。

用 `AgentContext` 做共享记忆，用 `AgentMessage` 做通信协议，一个下午就把四个Agent联调通了。

## Day 5：多模态 — 手机拍照秒识书架

**7月17日**

这是Demo的"高光时刻"。

Step 3.7 Flash 的原生多模态能力远超预期。我们给出一张标有"I247.5"的书架照片，模型直接返回：
```json
{"location": "2F 文学区推荐书架", "floor": "2F", "confidence": 0.89}
```

不需要单独的OCR、不需要YOLO检测、不需要额外训练——Flash 直接"看懂"了书架。

这个能力让我们的Demo脚本有了灵魂：**拍照→识别→导航**，一条链路打通，评委看一眼就能理解这个产品在做什么。

## Day 6-7：前端 — 让评委3秒看懂的界面

**7月18-19日**

技术再牛，评委看的是演示。所以前端界面只有一个原则：**移动端优先，3秒看懂**。

我们选择了原生HTML/CSS/JS——零构建步骤，浏览器直接打开就能用。不用React/Vue的原因很简单：没有时间折腾webpack配置。

移动端的相机调用是最大的坑。`getUserMedia` 在HTTP下不可用（需要HTTPS或localhost），还好我们准备了"从相册选择"的降级方案。CSS的动画效果调了好几版——打字指示器的三个小圆点跳动的节奏感，团队讨论了半小时。

## Day 8：Demo视频 — 90秒讲好一个故事

**7月20日**

视频脚本改了四版。最终定下的结构：

1. 0-15秒：打开页面，展示欢迎界面（"你好！我是吉小图👋"）
2. 15-40秒：文本提问"图书馆开放时间是？"→ 流式回复逐字出现
3. 40-70秒：拍照演示——对书架拍照→AI识别"2F文学区"
4. 70-90秒：导航——"怎么去期刊阅览室？"→ 分步路线上屏

90秒卡得很死，因为评委一天看几百个项目，超过2分钟的快进概率极大。

## Day 9：打磨 — 细节里藏着魔鬼

**7月21日**

全流程回归测试。发现了几个问题：
- 如果Step 3.7 Flash API超时怎么办？→ 加了关键词降级搜索
- 如果用户传了10MB+的图片怎么办？→ 加了大小限制和压缩提示
- 如果会话过期了怎么办？→ 加了TTL自动清理

加了降级策略后，系统的鲁棒性明显提升——**即使模型API挂了，最核心的FAQ查询和关键词匹配依然能用**。这也是评审中"项目完整性"（20%）的体现。
