# 技术栈说明文档

## 概述

本项目严格遵循 DGX Spark Hackathon 平台适配性要求（评审权重15%），深度使用 NVIDIA SDK、NVIDIA 开源模型及 Stepfun 阶跃星辰模型。以下逐一说明每个技术组件的具体用途和使用方式。

---

## 一、Stepfun 阶跃星辰模型（硬性要求）

### Step 3.7 Flash — 核心推理引擎

| 使用场景 | 具体用途 | 技术参数 |
|----------|---------|---------|
| **多模态理解** | 用户拍照上传书架/图书封面 → Flash 原生视觉理解，识别索书号标签、区域标识、图书信息 | temperature=0.3（保证识别准确度） |
| **对话生成** | 基于检索上下文和对话历史生成自然、准确的中文回复 | temperature=0.7, max_tokens=2048 |
| **意图分类** | 7类用户意图的精确分类（policy_query/location_where/navigate_to/book_lookup/opening_hours/facility_query/general_chat） | temperature=0.1（保证分类稳定） |
| **查询重写** | 将口语化查询（"那个文学的书在哪儿来着"）重写为检索友好的格式（"文学区 图书位置 2F"） | temperature=0.3 |
| **文本嵌入** | 为RAG知识库生成向量嵌入（1536维），用于ChromaDB语义检索 | 批量处理（20条/批） |
| **重排序** | 对向量检索Top-10结果进行LLM相关性打分，选出Top-3 | temperature=0.1 |

**选择 Flash 版本的原因**：Flash 模型在保持多模态理解和工具调用能力的同时，推理延迟更低，适合图书馆导览场景的实时交互需求（目标响应时间 < 3秒）。

---

## 二、NVIDIA 开源模型

### NVIDIA Nemotron-4 — 本地推理后备

| 使用场景 | 具体用途 |
|----------|---------|
| **Verify Agent 交叉校验** | 作为独立于 Step 3.7 Flash 的第二模型，对AI回答进行事实核查，降低幻觉风险 |
| **本地后备推理** | 当 Step 3.7 Flash API 不可用时（网络波动/限流），自动切换至本地 Nemotron 推理，保证服务连续性 |
| **敏感信息过滤** | 在对话层面进行额外的内容安全检查 |

---

## 三、NVIDIA SDK

### TensorRT-LLM — 推理加速

| 使用场景 | 具体用途 |
|----------|---------|
| **Nemotron 推理加速** | 将 Nemotron-4 模型通过 ONNX → TensorRT Engine 转换，实现 INT8 量化推理，延迟降低 40%+ |
| **Step 3.7 Flash 加速** | 在 DGX Spark 节点上使用 TensorRT-LLM 加速 Flash 模型的本地推理（如可用） |

### RAPIDS cuDF — GPU 加速数据处理

| 使用场景 | 具体用途 |
|----------|---------|
| **BM25 稀疏检索加速** | 使用 RAPIDS cuDF 在 GPU 上进行 TF-IDF 词频统计，加速混合检索中的 BM25 召回环节 |
| **批量嵌入后处理** | 对 Step 3.7 Flash 返回的嵌入向量进行 GPU 加速的归一化和预处理 |

---

## 四、其他关键组件

| 组件 | 用途 |
|------|------|
| **ChromaDB** | 轻量级向量数据库，存储知识库文档的嵌入向量，支持元数据过滤（按类型/楼层/类别） |
| **FastAPI** | 异步Python Web框架，提供 REST API 和 SSE 流式响应 |
| **PyMySQL** | 连接吉利学院图书馆 MySQL 馆藏数据库（只读），同步真实图书数据 |
